# ArcReel 本地调研与改造工作日志

日期：2026-05-15

## 背景目标

我们正在评估并尝试把 ArcReel 作为“AI 小说转动画工作流软件”的基础项目。目标产品的大方向是：让 AI 按章节读取小说，自动拆分工作流，生成角色设定、场景预设、分镜、首尾帧、视频片段、字幕、配音和最终成片，并通过可视化控制台管理任务、日志、素材库和 API 配置。

用户最初设想的是多 agent 协作架构：

- 总管 AI：负责原文切片、脚本分发、提示词编写、角色设定、场景预设、首尾帧、分镜、润色等编排。
- 员工 AI 1：生成角色参考图、风格概念图等，并保存到素材库。
- 员工 AI 2：基于提示词和参考图做角色形象设定，并支持主管 AI 调节权重。
- 员工 3：脚本输入和视频参数预设控制，包括视频时长、镜头数、景别、运镜、动作、光影等。
- 员工 AI 4：按时间轴、景别、运镜拆解分镜。
- 员工 AI 5：为每段分镜生成首尾帧提示词。
- 员工 AI 6：提示词润色优化。
- 员工 AI 7：视频生成。
- 控制台右侧任务面板：至少包含正在处理的任务、任务日志、文件归档、API 设置等。

## 已完成的判断

1. ArcReel 与目标方向高度接近。

   ArcReel 已经具备小说导入、角色/线索提取、剧集拆分、脚本 JSON、角色设计、分镜图、视频片段、素材库、任务队列、SSE 状态推送、用量统计、版本管理和剪映草稿导出等能力。它不是完全等同于目标产品，但比从零开始更接近。

2. ArcReel 当前强依赖 Claude Agent SDK。

   当前智能体运行时主要在 `server/agent_runtime/`，包括 `AssistantService`、`SessionManager`、`SessionActor`、transcript 读取、stream projector、turn grouper 等。内部大量逻辑直接绑定 Claude SDK 的 `ClaudeSDKClient`、`ClaudeAgentOptions`、`.claude/skills`、`Skill`、`Task`、`AskUserQuestion`、Claude transcript 格式和权限 hook。

3. OpenAI Agents SDK 可以作为未来替代方向，但不是简单换包。

   ArcReel 外层 API 可以保留，例如 `/api/v1/projects/{project}/assistant/...` 这一层只依赖 `AssistantService` 的抽象行为。但内部 Claude runtime 需要新增 OpenAI runtime 适配层，逐步复刻 session、streaming、tools、handoffs、history、权限和 UI event 映射。

4. 音频工作流需要后续独立设计。

   ArcReel 当前音频能力主要依赖视频生成模型自身的音频输出或提示词中的 `dialogue` / `ambiance_audio`。这对精准人物对白、角色声线一致性、字幕对齐和场景声控制不够可靠。后续更合理的方向是音画分离：静音视频 + TTS 人声 + SFX/环境音 + BGM + 字幕时间轴 + FFmpeg/剪映混音。

## 安装过程中遇到的问题

1. `uv sync` 曾在电脑突然关机时中断。

   处理建议是删除半成品 `.venv` 后重新执行：

   ```bash
   cd /mnt/c/Users/86188/Desktop/pindou
   rm -rf .venv
   uv sync
   ```

   后续解释过：`Resolved 124 packages` 是解析到的依赖总数，`Preparing packages... (0/16)` 是本次需要实际准备的包数量，很多包可能已在 uv 缓存中，不代表只安装了 16 个依赖。

2. `pnpm install` 在 `/mnt/c/.../frontend/node_modules` 下反复报 `ERR_PNPM_EACCES`。

   典型错误：

   ```text
   ERR_PNPM_EACCES importPackage ... EACCES: permission denied, rename ..._tmp -> ...
   ```

   判断原因：WSL 在 Windows 文件系统 `/mnt/c` 上大量创建、重命名 `node_modules` 文件时，容易遇到 DrvFs、Windows 文件锁、杀毒或权限问题。`pnpm store prune` 和 `--package-import-method=copy` 只能部分缓解，不能彻底解决。

3. 曾给出过折中方案：源码留在 Windows，`node_modules` 放到 WSL ext4。

   命令大意：

   ```bash
   cd /mnt/c/Users/86188/Desktop/pindou/frontend
   rm -rf node_modules
   mkdir -p ~/.arcreel-node_modules/frontend
   ln -s ~/.arcreel-node_modules/frontend node_modules
   pnpm install --package-import-method=copy --child-concurrency=1
   ```

   含义：`frontend/node_modules` 变成指向 `/home/czy/.arcreel-node_modules/frontend` 的软链接，真实依赖文件放在 WSL Linux 文件系统里。这样可以减少 `/mnt/c` 上的 EACCES 问题，但要求前端命令统一在 WSL 中运行，不要混用 Windows Node。

## VS Code WSL Remote 说明

用户已安装 VS Code 的 WSL 扩展。

要让 Codex/IDE 直接读写 WSL 路径，需要在 VS Code 的 WSL Remote 窗口里打开：

```bash
/home/czy/pindou
```

可以通过两种方式进入：

1. 在 WSL 终端执行：

   ```bash
   code ~/pindou
   ```

2. 在 Windows VS Code 中使用命令面板：

   ```text
   Remote-WSL: Connect to WSL using Distro...
   ```

   选择 `Ubuntu-24.04`，然后打开文件夹 `/home/czy/pindou`。

无论哪种方式，第一次都会在 WSL 中下载 VS Code Server：

```bash
~/.vscode-server
```

这是正常行为。它不是在 WSL 中安装完整图形版 VS Code，而是安装无界面的远程后台服务，用来让 Windows VS Code 读写 WSL 文件、运行终端、运行语言服务和工作区扩展。

如果 Codex 是 VS Code 扩展，则需要在 WSL Remote 窗口中点：

```text
Install in WSL: Ubuntu-24.04
```

这样新开的 Codex 会话才会绑定 `/home/czy/pindou`，而不是当前 Windows 工作区。

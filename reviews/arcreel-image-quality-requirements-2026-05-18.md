# ArcReel 回归改造需求：先解决出图质量

日期：2026-05-18

## 一句话结论

后续不再优先从 0 写一个新软件，而是回到 ArcReel 上改造。ArcReel 当前的 Agent/Skill 工作流模式可以保留，主要问题不是“工作流方向错了”，而是图像生成质量差、提示词链路不透明、角色资料包设计不适合稳定生成短视频画面。

第一阶段目标：先把图像生成做稳定、可查、可控。

## 当前最痛的问题

1. 出图质量差，尤其是角色图和分镜图。
2. 不知道差在哪里：可能是角色描述、风格模板、prompt 包装、参考图策略、模型参数或后端返回问题。
3. 同样的角色描述直接拿到 Vidu 网页端生成，效果反而还可以，说明 ArcReel 内部传给图片模型的最终内容很可能做了额外包装，且包装方式可能破坏了结果。
4. 当前用户看不到完整最终 prompt，也看不到模型请求参数、参考图列表、原始返回和失败原因。
5. 角色设定图现在更像“四格角色设定图”，这会污染后续分镜参考，容易导致多头、多身体、多人物、构图混乱。

## 已确认保留的现状

以下部分先不大改：

1. 继续基于 ArcReel，而不是继续开发从 0 写的 `czyct`。
2. 保留 ArcReel 的 Agent/Skill 工作流模式。
3. 保留当前从小说导入、生成剧集脚本、生成分镜 JSON、生成资产、生成分镜图、图生视频的大流程。
4. 先聚焦“图生视频 / storyboard 模式”。
5. 模型供应商先不作为改造重点，用户已经配置 Vidu。
6. 场景、道具先保持现状，不作为第一阶段重点。
7. 暂不做音频、TTS、字幕、音画同步。
8. 前端主界面大改先放后面，只做必要的状态可见性和 prompt 可见性。

## 第一阶段优先级

### P0：生成链路透明化

必须能看到每一次生成到底喂给模型了什么。

每个生成任务都应该保存：

1. 任务类型：角色主参考图、角色三视图、场景图、道具图、分镜首帧、视频等。
2. 原始业务输入：角色描述、场景描述、分镜 JSON、风格模板、用户补充要求。
3. 最终 prompt：真正发送给模型的完整文本。
4. 参考图列表：传入模型的所有图片路径、用途、顺序。
5. 模型参数：provider、model、aspect ratio、resolution、duration 等。
6. 原始返回：模型返回的 task id、图片 URL、本地保存路径、错误信息。
7. 失败与重试记录：尤其是 504、超时、被拒、解析失败。

建议在项目目录内保存调试文件，例如：

```text
projects/{project_id}/debug/generation_logs/
  2026-05-18Txxxx_character_fullbody_苏洄.json
  2026-05-18Txxxx_storyboard_E1S01.json
```

前端后续可以加“查看最终 prompt / 查看请求详情 / 查看原始返回”按钮，但第一步可以先落盘。

### P0：角色资料包重做

当前不要继续把“角色设定图”做成横版四格图。

目标角色资料包结构：

1. 一个角色可以有多个造型 / 形态。
2. 每个造型先生成一张“单人全身主参考图”。
3. 再基于“单人全身主参考图”生成一张“三视图”。
4. 后续分镜优先引用“单人全身主参考图”，不要默认引用四格图或多面板图。
5. 三视图用于人工审阅和后续一致性约束，不应直接污染分镜构图。

建议结构：

```json
{
  "characters": {
    "苏洄": {
      "description": "...",
      "forms": {
        "default": {
          "label": "默认造型",
          "description": "...",
          "main_full_body": "characters/苏洄/default/full_body.png",
          "turnaround": "characters/苏洄/default/turnaround.png",
          "notes": ""
        },
        "hotel_sick": {
          "label": "酒店病弱造型",
          "description": "...",
          "main_full_body": "characters/苏洄/hotel_sick/full_body.png",
          "turnaround": "characters/苏洄/hotel_sick/turnaround.png",
          "notes": ""
        }
      }
    }
  }
}
```

第一阶段先支持最小版本：

1. 每个角色至少一个 `default` form。
2. 每个 form 至少有 `main_full_body` 和 `turnaround` 两个槽位。
3. 旧字段 `character_sheet` 可以暂时保留兼容，但新流程不再优先使用它。

### P0：拆分图片 prompt 类型

不能继续所有图片任务共用一套资产设计图 prompt。至少需要拆成：

1. 角色单人全身主参考图 prompt。
2. 角色三视图 prompt。
3. 角色某一造型 / 服装变体 prompt。
4. 场景主参考图 prompt。
5. 道具参考图 prompt。
6. 分镜首帧 prompt。
7. 视频 prompt。

重点区别：

角色主参考图要强调：

1. 单人。
2. 全身。
3. 站姿清晰。
4. 纯净背景。
5. 不要多面板。
6. 不要文字。
7. 不要重复人物。
8. 不要多个头或多个身体。

分镜首帧要强调：

1. 这是一个镜头画面，不是设定图。
2. 只能出现当前分镜需要的人物。
3. 按分镜的景别、构图、光影、动作、情绪生成。
4. 参考角色图只用于角色一致性，不要照搬参考图的多面板布局。
5. 参考场景图只用于环境一致性，不要生成平铺设定图。

### P0：参考图策略重做

当前风险：分镜生成会收集角色图、场景图、道具图、上一张分镜图等作为参考。如果角色图本身是四格图，分镜模型可能把“四格/多视角/多人物”误认为目标画面，导致多头、多身体、画面混乱。

第一阶段要求：

1. 分镜生成默认只引用角色的 `main_full_body`，不要引用四格角色图。
2. 三视图默认不进入分镜生成，除非用户明确选择。
3. 多角色分镜要限制参考图数量和用途，避免模型把参考图当成拼图任务。
4. 场景图如果是“主图 + 细节小图”的多面板，也可能污染分镜；第一阶段可先少用或只作为弱参考。
5. 每次任务日志必须记录到底用了哪些参考图。

## 当前代码观察

基于当前仓库初步观察，角色图质量问题有明确嫌疑点。

`lib/prompt_builders.py` 里的角色 prompt 当前强制：

```text
横版 16:9 四格布局，纯白背景：左侧约 40% 宽为胸像特写，右侧三个等宽面板分别为正面 / 四分之三侧面 / 背面的 A-Pose 全身视图。
```

这说明 ArcReel 不是把“角色描述”原样发给 Vidu，而是在外面包了一层“四格角色设定图”要求。用户在 Vidu 网页直接发角色描述效果更好，很可能就是因为网页端没有这层多面板包装。

测试项目：

```text
projects/2-4932a2d8
```

关键信息：

1. `generation_mode` 是 `storyboard`。
2. 图像模型为 `vidu/viduq2`。
3. 风格模板为 `anim_kr_webtoon`。
4. 剧集脚本位于 `projects/2-4932a2d8/scripts/episode_1.json`。
5. 中间稿位于 `projects/2-4932a2d8/drafts/episode_1/step1_normalized_script.md`。
6. 已生成角色图、场景图、分镜图，可以作为后续排查样本。

## 剧本和分镜的需求判断

现在 ArcReel 已经不是简单“小说硬切片”，它已经有中间稿、剧集脚本、分镜 JSON。问题不是完全没有这一步，而是质量还不够稳定。

后续改造方向：

1. 保留“小说 -> 中间稿 -> 剧集脚本 -> 分镜 JSON”的链路。
2. 增加 `story_bible`，让人物关系、外貌、场景风格、禁忌项先固定下来。
3. 剧集脚本和分镜 JSON 都要可编辑。
4. 分镜时长支持两种模式：
   - 用户指定目标时长。
   - 按剧情自动分配。
5. 分镜 prompt 要更像导演给美术和摄影的说明，而不是只描述剧情。

## 需要新增或暴露的能力

### 1. Prompt 调试面板

先不要求做复杂 UI，但最终应该能在每个资产 / 分镜 / 视频任务上看到：

1. 最终 prompt。
2. 风格模板。
3. 引用的角色资料。
4. 引用的场景资料。
5. 参考图。
6. 模型参数。
7. 原始返回。

### 2. 生成质量对比

同一个角色或分镜，最好能保留多次生成版本，方便比较：

1. 当前版本。
2. 历史版本。
3. 使用的 prompt。
4. 使用的参考图。
5. 用户是否标记为可用。

ArcReel 已经有版本目录，可以优先接入现有版本机制。

### 3. Prompt Builder 分层

建议把 prompt builder 分成两层：

1. 业务层：根据任务类型组织角色、场景、分镜、风格、镜头语言。
2. 模型适配层：根据 Vidu 等模型特点控制长度、参考图数量、负面约束、比例参数。

第一阶段不需要换模型，但要避免把所有模型都当成同一种 prompt 接口。

## 暂缓事项

以下事项不是当前第一阶段重点：

1. 大规模 UI 重做。
2. OpenAI Agents SDK Runtime。
3. Claude Runtime 抽象。
4. 多 Agent 架构重做。
5. 场景资料包深度重做。
6. 道具资料包深度重做。
7. 音频、TTS、字幕、音画同步。
8. 成本统计。
9. 更换 Vidu 或重新评估模型能力。

## 下一步建议

建议下一步直接做一次“图像链路体检”，先不改大架构：

1. 给角色生成、场景生成、分镜生成、视频生成补最终请求日志。
2. 用 `projects/2-4932a2d8` 复现一次角色图和分镜图生成。
3. 对比日志里的最终 prompt 与用户直接发给 Vidu 网页的 prompt。
4. 把角色 prompt 从四格图改成“单人全身主参考图”。
5. 让分镜生成优先引用单人全身主参考图，而不是旧 `character_sheet`。
6. 再观察分镜图是否明显改善。

如果这一步能改善，说明主要问题在 ArcReel 的 prompt 包装和参考图策略；如果仍然差，再继续查 Vidu API 参数、参考图上传方式、分辨率、比例和后端返回。
